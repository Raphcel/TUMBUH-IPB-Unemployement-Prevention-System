# Database

TUMBUH uses PostgreSQL as the system of record. SQLAlchemy models and Alembic migrations are the schema authority.

## Database Modes

| Mode | Use case | Connection |
|---|---|---|
| Local Docker PostgreSQL | Isolated local development | `postgresql://postgres:postgres@localhost:5433/career_tracker` |
| Shared Supabase PostgreSQL | Team dev/staging | Supabase Session Pooler URL with `?sslmode=require` |
| Production Supabase PostgreSQL | Deployed backend | Direct connection if IPv6 is available; otherwise Session Pooler |

## Core Tables

| Table | Purpose |
|---|---|
| `users` | Students, HR staff, and admins |
| `companies` | Employer profiles |
| `opportunities` | Job and internship listings |
| `applications` | Student applications to opportunities |
| `bookmarks` | Student saved opportunities |
| `externships` | Student-recorded independent externships |

## Relationships

- A company can own many opportunities.
- A student can create many applications.
- An opportunity can have many applications.
- A student can bookmark many opportunities.
- A student can create many externship records.
- HR users are associated with company management workflows.

## Migrations

Run migrations from `be-web/`:

```bash
alembic current
alembic upgrade head
alembic history
```

Generate a migration after model changes:

```bash
alembic revision --autogenerate -m "describe change"
```

Review autogenerated migrations before applying them. Do not use the Supabase dashboard for normal schema changes unless the change is later reconciled into Alembic.

## Seed Data

After migrations:

```bash
python -m scripts.seed
```

Seed data provides demo accounts and sample business records for local QA. See [Testing](testing.md) for demo credentials and manual scenarios.

## Inspection

Using `psql` with the local Docker database:

```bash
psql postgresql://postgres:postgres@localhost:5433/career_tracker
```

Useful checks:

```sql
\dt
select count(*) from users;
select count(*) from companies;
select count(*) from opportunities;
select count(*) from applications;
```

Swagger UI at `http://127.0.0.1:8000/docs` is also useful for validating database-backed API behavior.

## Reset Warning

Dropping tables, deleting Docker volumes, or re-seeding shared databases can destroy data. Only reset local disposable databases unless the whole team has agreed.
